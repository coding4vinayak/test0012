# Six viral AI micro-SaaS apps via cloned Express scaffold

Six standalone Node.js/Express apps (website roaster, excuse generator, code roaster, personality-from-tweets analyzer, startup idea generator, resume roaster) each following the same pattern: Express + EJS + better-sqlite3 + bcrypt + express-session, with mock AI generation from template pools, shareable output via unique IDs, and optional auth.

**Watch for:** Stored XSS via the `language` parameter in the code roaster's client-side rendering (confirmed). Port collision between ai-roast-my-resume and ai-startup-idea-generator, both defaulting to 3045 (confirmed). No rate limiting on generation endpoints that write to the database (confirmed). Session cookies missing `secure` flag across all six apps (confirmed).

## High-level view

All six apps share an identical Express scaffold: session middleware with a hardcoded fallback secret (warned at startup), EJS views with escaped output, better-sqlite3 on WAL mode, and a conditional-listen pattern for testability. The architectural repetition means a bug in the pattern is a bug in all six apps, but it also means fixes propagate easily.

The generation endpoints accept unauthenticated POST requests and write results directly to SQLite. With no rate limiting or input size constraints, a single client can fill the database with unbounded rows and arbitrarily large payloads (the code roaster and resume roaster store raw user text with no upper bound beyond Express's default 100KB JSON limit).

Client-side rendering uses `innerHTML` with template literals that interpolate server response data. The server-generated roast text includes the user-supplied `language` field (code roaster) without sanitization. The server-rendered share pages use EJS `<%= %>` (auto-escaped) so the XSS only manifests in the SPA-style client rendering path.

Session cookies are configured with only `maxAge`. The `secure` flag is not set, meaning cookies transmit over plain HTTP in production. No CSRF protection exists, but since the generation endpoints accept JSON bodies (not form posts), the practical CSRF surface is limited — modern browsers block cross-origin JSON POSTs with credentials via CORS preflight.

## Stored XSS in the code roaster's client-side rendering

The code roaster's `generateCodeRoast` function interpolates the user-supplied `language` parameter directly into verdict template strings:

```javascript
// routes/roast.js line 165
`Overall, this ${language} code is what happens when...`
```

The `language` value arrives from `req.body.language` with no sanitization. The route stores the assembled roast text in SQLite and returns it in the JSON response. The client's `displayResults` function then injects this text via `innerHTML`:

```javascript
// public/js/app.js
resultContainer.innerHTML = `...
  <p>${data.roast.verdict}</p>
...`;
```

A payload like `<img src=x onerror=alert(document.cookie)>` as the language value would execute when the result is displayed. The server-rendered share page (`share.ejs`) uses EJS `<%= %>` which auto-escapes, so this XSS is limited to the initial client rendering — but that is still the primary user-facing interaction.

The fix is either to whitelist the language parameter against a known set (which also improves roast relevance) or to escape HTML entities before interpolation.

## Port 3045 collision

`ai-roast-my-resume/server.js` and `ai-startup-idea-generator/server.js` both declare `const PORT = process.env.PORT || 3045`. This prevents running both apps simultaneously in development without manually overriding one via environment variable. The remaining four apps use distinct ports (3040-3044).

## Unbounded writes and missing rate limits

Every generation endpoint (`/roast/generate`, `/excuse/generate`, `/analyze/generate`, `/idea/generate`) accepts unauthenticated requests and writes a new row to SQLite on success. There is no per-IP throttling, no session-based quota, and no captcha. For the code roaster and resume roaster, the raw user text (code snippet, resume text) is stored verbatim with no upper-bound check beyond Express's default 100KB body limit.

The resume roaster enforces a minimum length (50 characters) but no maximum. The code roaster checks only for empty input. An attacker could POST 100KB payloads in a loop to exhaust disk space on the SQLite file. The `better-sqlite3` synchronous writes also mean these endpoints block the event loop during insertion, making a flood doubly effective as a denial-of-service vector.

## Session cookie posture

The session configuration across all six apps:

```javascript
app.use(session({
  secret: process.env.SESSION_SECRET || 'ai-<app-name>-secret-key',
  resave: false,
  saveUninitialized: false,
  cookie: { maxAge: 24 * 60 * 60 * 1000 }
}));
```

The `secure` flag is not set, so cookies will transmit over unencrypted connections in production. The `sameSite` attribute is also unset (defaults to `Lax` in modern browsers, which provides baseline CSRF protection for form submissions). The fallback secrets are unique per app but predictable (they contain the app name).

## Test coverage gaps

Not tested: maximum input sizes, the client-side JavaScript rendering path where the XSS lives, and any concurrent-request behavior. Each app has 22-27 integration tests covering auth flows, generation endpoints, share pages, and dashboard rendering — but all through the HTTP/JSON interface, never exercising the browser-side `innerHTML` path.
